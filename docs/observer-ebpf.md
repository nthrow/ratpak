# `internal/observer/ebpf`

The eBPF-backed implementation of the filesystem observer.

## Files

- `openat.bpf.c` — kernel-side program. Compiled to BPF bytecode by clang at `go generate` time and embedded into the Go binary.
- `observer.go` — userspace-side. Loads the program, attaches tracepoints, decodes the ring buffer, exposes `AddRoot` for seeding the tracked set, emits `observer.Event`s.
- `openat_<arch>_bpfel.{go,o}` — generated by `bpf2go`; `.gitignored`.

## Kernel side (`openat.bpf.c`)

Three maps and six programs.

### Maps

- `events` — `BPF_MAP_TYPE_RINGBUF`, 1 MiB. The kernel→userspace event channel.
- `pending` — `BPF_MAP_TYPE_LRU_HASH`, 8192 entries, key = tid (u32), value = userspace pointer (u64). Per-thread scratch used to pass the filename argument from the enter program to the exit program.
- `tracked_pids` — `BPF_MAP_TYPE_LRU_HASH`, 16384 entries, key = tid (u32), value = u32 marker. The kernel-side membership filter: only events whose calling tid is in this map flow through to userspace.

### Programs

Syscall hooks (filtered on `tracked_pids`):

- `trace_openat_enter` (tp/syscalls/sys_enter_openat)
- `trace_openat_exit`  (tp/syscalls/sys_exit_openat)
- `trace_openat2_enter` (tp/syscalls/sys_enter_openat2)
- `trace_openat2_exit`  (tp/syscalls/sys_exit_openat2)

Membership maintenance:

- `trace_sched_fork`  (tp/sched/sched_process_fork)
- `trace_sched_exit`  (tp/sched/sched_process_exit)

Both `_enter` programs check `is_tracked()` first; if yes, stash the filename pointer (arg index 1) keyed by tid in `pending`. Both `_exit` programs do the same check, then look up the stashed pointer and, if `ctx->ret >= 0`, copy `{pid, tgid, comm, filename}` into a ringbuf record. The LRU policy means an evicted-mid-syscall stash is safe.

`trace_sched_fork` runs in the parent's context. If `parent_pid` is in `tracked_pids`, `child_pid` is added — propagating membership across both process and thread clones (since both are by-TID in this map).

`trace_sched_exit` removes the exiting task's tid from `tracked_pids`.

The tracepoint context structs (`sys_enter_args`, `sys_exit_args`, `sched_fork_args`, `sched_exit_args`) are hand-defined to match the kernel's stable per-tracepoint format file layout, so we don't need vmlinux.h or BTF.

### Important: tracepoint context layouts vary

Each tracepoint's context layout is whatever its `format` file says. Don't assume same-family tracepoints share the same layout — they often don't. Two examples in this program:

- `sched/sched_process_fork`: `comm` fields are `__data_loc char[]` (4-byte offset/length references, NOT inline strings). Total context is 24 bytes. `parent_pid` is at offset 12, `child_pid` at offset 20.
- `sched/sched_process_exit`: `comm` is inline `char[16]`. `pid` is at offset 24.

Same `sched/` group, completely different shapes. Always confirm by reading `/sys/kernel/tracing/events/<group>/<name>/format` for the kernel you're targeting before defining the context struct. The verifier will reject attach (often with a misleading `permission denied` rather than a verifier-style error) if a struct field access reads past the actual context size.

### Why track by TID rather than TGID

The fork tracepoint exposes child_pid (= the new task's TID), but not its TGID. For thread clones, child_pid is a new TID belonging to the parent's TGID; for process forks, child_pid equals the new TGID. Tracking by TID handles both uniformly: every clone propagates membership directly, and the openat hooks filter by the calling thread's TID via `bpf_get_current_pid_tgid() & 0xffffffff`. No vmlinux/BTF read of `task_struct->tgid` required.

The downside: a multi-threaded process whose threads existed before `AddRoot` was called won't have all its threads in the map. Userspace seeding handles this by walking `/proc/<pid>/task/` when retro-attaching to an already-running process. For the current `observe` flow (seed immediately after `cmd.Start`), the launched flatpak run is single-threaded at seed time, so propagation via fork is sufficient.

## Userspace side (`observer.go`)

Generated bindings via:

```go
//go:generate go run github.com/cilium/ebpf/cmd/bpf2go -cc clang -cflags "-O2 -g -Wall" -target amd64,arm64 openat openat.bpf.c
```

`bpf2go` builds a `*_bpfel.o` per target arch (amd64, arm64) and an embedding `.go` file that wraps each one.

`Observe` does:

1. `rlimit.RemoveMemlock()` — soft, only fatal on pre-5.11 kernels.
2. `loadOpenatObjects` — load the embedded BPF object into the kernel.
3. Attach all six tracepoints (four syscall + two sched) via `link.Tracepoint`.
4. Open a ringbuf reader on the `events` map; pin the loaded objects on the Observer so `AddRoot` can write to `tracked_pids`.
5. Spawn a goroutine that reads records, decodes each into a `rawEvent`, and emits an `observer.Event` on the returned channel.

`AddRoot(pid int)` writes `(pid, 1)` into `tracked_pids`. Until at least one `AddRoot` call lands, the kernel-side filter drops every event — the channel will look idle.

Cancellation: when ctx fires, the reader is closed; the goroutine clears the pinned objects and tears down links + maps.

## `rawEvent` ↔ C `struct event`

```c
struct event {
    __u32 pid;
    __u32 tgid;
    char  comm[16];
    char  path[256];
};
```

```go
type rawEvent struct {
    PID  uint32
    TGID uint32
    Comm [16]byte
    Path [256]byte
}
```

Decoded via `binary.Read` with `binary.NativeEndian`. The Go side trims each `[N]byte` field at the first NUL.

## Privileges

Loading BPF programs and attaching tracepoints requires `CAP_BPF` and `CAP_PERFMON` on modern kernels (≥ 5.8), or `CAP_SYS_ADMIN` on older ones. `make setcap` grants these to `bin/ratpak` so day-to-day `observe` (and the future daemon) runs as the regular user.
